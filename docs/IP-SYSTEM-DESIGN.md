# IP 게임 시스템 설계서

## 0. 한 줄 정의

> **IP = 게임 시스템(엔진) 그 자체.**
> 시스템은 고정 자산이고, 그 위에 **장르(RPG↔방치형)** 와 **컨셉(테마)** 만 갈아끼워
> 여러 게임으로 운영한다.

캐릭터·세계관이 IP가 아니다. **성장·전투·경제 규칙**이 IP다.
따라서 한 번 잘 만든 시스템을 판타지 RPG로도, SF 방치형으로도 재출시할 수 있다.

## 1. 3-레이어 아키텍처

```
┌─ Core System (IP · 고정) ───────────────────────────┐
│  장르도 컨셉도 모른다. 순수 규칙과 데이터.               │
│    · archetypes  유닛 역할(방어/공격/지원)              │
│    · stats       레벨·랭크 성장 + 모디파이어 반영         │
│    · modifiers   성장요소 합산(스킬·강화·장비)           │
│    · skills      스킬 카탈로그/장착/강화                 │
│    · enhance     강화(각인) 노드 정의                    │
│    · gear        장비 카탈로그/제작/장착/강화             │
│    · gacha       소환(확률·천장) — RNG 주입              │
│    · rng         시드 가능 난수                          │
│    · character   레벨업·돌파·장착·강화 액션(자원 소모)    │
│    · units       유닛 인스턴스 → 전투 프로필              │
│    · resolution  전투 판정 엔진  → { win, duration }    │
│    · progression 무한 스테이지 사다리                    │
│    · economy     추상 자원(currency/growth/summon)     │
│    · gameState   세이브(보유·진행·지갑) — 장르/컨셉 없음  │
├─ Genre Layer (Driver · 스왑) ───────────────────────┤
│  "시간을 누가 굴리나" 만 다르다. 같은 resolve()를 쓴다.   │
│    · rpg   플레이어가 battle() 트리거, 행동력 소모, 능동   │
│    · idle  tick(dt) 자동 반복·누적, 오프라인·환생, 수동0  │
├─ Concept Layer (Skin · 스왑) ───────────────────────┤
│  숫자는 그대로. 이름·이모지·색상만 매핑.                  │
│    · fantasy  VANGUARD→수호기사, currency→골드           │
│    · scifi    VANGUARD→가디언프레임, currency→크레딧       │
└─────────────────────────────────────────────────────┘
```

의존 방향은 **한 방향뿐**: `concept → genre → core`.
core 는 위 레이어를 절대 import 하지 않는다 → 그래서 재사용된다.

## 2. 핵심 설계: 판정 엔진이 두 장르를 잇는다

`resolution.resolve(party, challenge)` 는 결정론적으로 다음을 반환한다:

```js
{ win: boolean, duration: number /* 클리어 소요 초 */ }
```

- **RPG** 는 `win` 을 쓴다 → 이기면 다음 스테이지 개방(진행 게이팅).
- **방치형** 은 `duration` 을 쓴다 → `dt / duration` 으로 초당 클리어 수·보상 누적.

> 하나의 순수 함수가 능동형과 자동형을 동시에 지탱한다. 이게 이 IP의 심장이다.

판정 모델(요약): 각 진영을 `HP · 유효DPS` 로 환산해 상호 처치 시간을 비교.
방어(def)는 `100/(100+def)` 피해 감쇠, 지원형은 팀 공격력 버프를 더한다.
스킬 효과(치명타·흡혈·관통·팀버프)도 여기서 반영된다(아래 참조).

## 3. 캐릭터 성장 — "같은 캐릭터를 다르게"

같은 원형(archetype)·같은 레벨이어도 **어떻게 키우느냐**로 유닛을 분화시킨다.
세 축이 서로 다른 종류의 차별화를 만든다:

| 축 | 성격 | 자원 | 효과 |
|---|---|---|---|
| **레벨업** | 균일 성장 | growth | 전 스탯 +8%/레벨. 상한 = 랭크×20 |
| **돌파(랭크)** | 한계 개방 | summon | 스탯 +25%/랭크 + 레벨 상한↑ + 스킬 슬롯↑ |
| **스킬 장착** | 선택 분화 | (장착 무료) | 슬롯에 스킬을 골라 끼움 → 빌드/역할 |
| **스킬 강화** | 선택 심화 | growth | 스킬 레벨 +15%/레벨 |
| **강화(각인)** | 방향 투자 | currency | 특정 스탯 노드에 집중(atk/hp/def/crit) |
| **장비** | 착용 성장 | currency | 슬롯별 flat 스탯 + 효과, 강화로 +12%/레벨 |

### 3-1. 모디파이어 파이프라인 (`modifiers.mjs`)

성장 요소가 흩어져 있으면 스탯 계산이 지옥이 된다. 그래서 **모든 소스를 한 곳에서 합산**한다:

```
collectUnitModifiers(unit)  →
  { statPct: {atk,hp,def,spd},                  // % 스탯 증가
    statFlat:{atk,hp,def,spd},                  // 고정 스탯 (장비)
    effect:  {critChance,critDamage,lifesteal,defPierce},  // 전투 효과
    teamBuff:{atk} }                             // 팀 버프
```

소스는 4개: **원형 버프 · 장착 스킬(레벨 반영) · 강화 노드 · 장착 장비**.
`stats.mjs`(스탯)와 `resolution.mjs`(판정)가 이 합산 결과를 공유한다.
새 성장 요소(예: 유물/각성)를 붙여도 여기 소스만 추가하면 된다.

### 3-2. 스탯 계산 순서 (`stats.mjs`)

```
1) 기본스탯 × 레벨배수 × 랭크배수      (원형 성장 — 균일)
2) × (1 + statPct)                    (스킬·각인 — 방향성)
3) + statFlat                         (장비 — 착용)
```

### 3-3. 스킬 전투 효과가 판정에 들어가는 법

| 효과 | 판정 반영 |
|---|---|
| 치명타 crit | dps × (1 + 확률×치명피해) — 프로필 단계 |
| 흡혈 lifesteal | 파티 실효HP × (1+흡혈), 상한 60% |
| 관통 defPierce | 적 방어 × (1−관통), 파티 내 최댓값 |
| 팀버프 teamBuff | 파티 dps × (1+합산) |

### 3-4. 검증 (`node system/demo-build.mjs`)

동일한 STRIKER Lv.40/R2 두 유닛을 다르게 키운 결과:

| 빌드 | 구성 | HP | DPS | 스테이지12 |
|---|---|---|---|---|
| A 글래스캐논 | 광폭+정밀, atk·crit 각인 | 3090 | 2716 | 3.2초 클리어 |
| B 브루저 | 요새+흡혈, hp·def 각인 | 4720(실효6136) | 1473 | 5.9초 클리어 |

> 출발점이 완전히 같아도, 투자 방향이 유닛의 정체성을 바꾼다.

### 3-5. 소환(가차) — 유닛 획득 (`gacha.mjs` + `rng.mjs`)

- **등급 확률**: N 50 / R 33 / SR 14 / SSR 3 (가중치). 등급이 시작 랭크를 정함(전설=R3).
- **역할**: VANGUARD/STRIKER/SUPPORT 균등.
- **천장(pity)**: 90회 안에 SSR 보장, SSR 획득 시 리셋.
- **10연차**: 최소 1개 SR 이상 보장.
- **RNG 주입**: `makeRng(seed)` 로 재현 가능 → 확률 검증·밸런싱에 사용.

### 3-6. 장비 — 착용 성장 (`gear.mjs`)

- 슬롯 3종: **weapon(공격) · armor(생존) · accessory(효과/속도)**.
- 장비는 flat 스탯 + 전투 효과(치명/흡혈/관통)를 준다 → modifiers의 한 소스.
- **제작(craft)** → 인벤토리, **장착(equip)** → 슬롯, **강화(enhance)** → flat +12%/레벨.
- 강화(각인)이 "본체 %투자"라면, 장비는 "착용 flat + 효과"로 역할이 갈린다.

검증(`node system/demo-gacha.mjs`): 10연차(천장·바닥보장 동작) 후,
뽑은 유닛에 장비 3종 장착·강화 → 전투력 347 → 679.

## 4. 레이어별 스왑 규칙

| 바꾸는 것 | 건드리는 파일 | core 변경 | 예시 |
|---|---|---|---|
| **장르** | `genres/*.mjs` 추가/교체 | ❌ 없음 | RPG → 방치형 |
| **컨셉** | `concepts/*.mjs` 추가/교체 | ❌ 없음 | 판타지 → SF |
| **밸런스** | `core/*` 상수 | ⭕ 공통 반영 | 성장률, 적 강화율 |

새 장르(예: 실시간 PvP)나 새 컨셉(예: 좀비 아포칼립스)을 붙일 때
**core 는 한 줄도 수정하지 않는다.** 파일 하나만 추가하면 된다.

## 8. 밸런스 시뮬레이터 (`system/sim/`)

문서가 "미정"으로 남긴 **숫자 감각**을 코드로 검증한다. 합리적 오토플레이어(자원을
벌어 재투자하는 가상 유저)를 7일간 돌려 성장 곡선·병목·해금 페이싱을 리포트한다.

- **`node system/sim/balance.mjs`** — 7일 곡선 + 병목 검출 + 해금 게이트 페이싱 +
  튜닝 실험(밸런스 상수를 바꿔 재시뮬레이션 비교)
- **`node system/sim/report.mjs`** — 곡선을 자체완결 HTML(인라인 SVG)로 시각화

### 진단 → 수정 → 재검증

1. **지수적 벽(진단)**: Day1에 ~30층 폭발 후 하루 +1. 파워가 **레벨 상한(랭크×20)**에
   묶이고, 유일한 돌파 수단인 환생이 **수입만** 늘리고 파워는 안 늘려 1.14ⁿ 난이도를
   못 따라감 → 최저달성률 45%로 뒤처짐, 병목 6회.
2. **환생이 장식용이었다(원인)**: `state.prestige`가 포인트만 쌓고 아무 배수도 미적용.
3. **수정**: 환생에 **상한 없는 글로벌 파워 배수**(`prestigePowerBonus`) 추가 +
   환생 시 회차 리셋(진행도 `peakStage`는 유지)하는 정통 방치형 루프.
   벽에서 환생 → 배수로 더 깊이 재등반.
4. **재검증**: 벽 해소. 36층→53층, 최저달성률 45%→91%, 병목 6회→**0회**(급정체 없음).

> 시뮬레이터가 문제를 찾고 → Core를 고치고 → 같은 시뮬레이터로 해소를 확인하는
> 폐루프. 아트 한 장 그리기 전에 밸런스가 굴러가는지 증명한다.

### 설계 산물: `core/balance.mjs`

모든 밸런스 상수(적/보상/성장/비용/환생)를 **단일 소스**로 모았다. 시뮬레이터가
`withBalance(override, fn)`로 값을 바꿔가며 실험하고, 게임 코드는 이 값을 참조만 한다.
새 성장·비용 곡선을 붙여도 한 곳만 수정하면 전 시스템에 반영된다.
